# How To Run: Adaptive Bypass Cache Replacement

This repository evaluates a custom **Adaptive Bypass Cache Replacement Policy** inside the **gem5** architectural simulator using the NAS Parallel Benchmarks (NPB).

This guide walks through exactly how to build the simulator, compile the benchmarks, and dispatch the evaluation pipelines on a SLURM high-performance cluster.

---

## 1. Building the gem5 Simulator
Because we have modified the internal C++ logic of gem5 (specifically `adaptive_bypass_rp.cc` and `base_set_assoc.hh`), you must re-compile the gem5 binary before running evaluations.

We provide a dedicated SLURM job script to build the simulator quickly using 8 cores:
```bash
# From the root of the project repository:
sbatch slurm_evaluation/build.sh
```
---

## 2. Compiling the NPB Benchmarks
Our evaluation specifically relies on statically compiled, single-threaded (SER) execution to produce accurate cache behavior. The benchmarks we evaluate are **IS** (Integer Sort), **CG** (Conjugate Gradient), **MG** (Multi-Grid), and **BT** (Block Tridiagonal).

1. Navigate to the NPB source directory:
   ```bash
   cd NPB3.3.1/NPB3.3-SER/
   ```
2. Compile the "Class S" size for all four benchmarks:
   ```bash
   make is CLASS=S
   make cg CLASS=S
   make mg CLASS=S
   make bt CLASS=S
   make lu CLASS=S
   make ft CLASS=S
   ```
3. The aggressively optimized static binaries will be placed in `NPB3.3.1/NPB3.3-SER/bin/`. Return to the project root:
   ```bash
   cd ../../
   ```

---

## 3. Launching the Evaluations
We provide fully-automated python and SLURM wrappers that evaluate the compiled benchmarks across a massive matrix of configurations. 

Specifically, they evaluate each benchmark against:
* **L2 Cache Sizes**: `128kB`, `256kB`, `512kB`, `1MB`
* **L2 Cache Associativity**: `8-way` 
* **Replacement Policies**: `LRURP` vs `AdaptiveBypassRP`
* **Initial Bypass Probabilities**: `0%`, `20%`, `50%`, `80%`, `100%`

To execute an evaluation script, you must first navigate *into* the evaluation directory, and then submit the script to the SLURM scheduler:
```bash
cd slurm_evaluation/

# Run the Integer Sort evaluation
sbatch run_is_eval.sh

# Run the Conjugate Gradient evaluation
sbatch run_cg_eval.sh

# Run the Multi-Grid evaluation
sbatch run_mg_eval.sh

# Run the Block Tridiagonal evaluation
sbatch run_bt_eval.sh

# Run the Fourier Transform evaluation
sbatch run_ft_eval.sh

# Run the Lower-Upper Gauss-Seidel evaluation
sbatch run_lu_eval.sh
```

### Where are results saved?
1. **gem5 Simulation Statistics**: The actual hardware statistics and traces (the output generated by gem5 itself for each specific config) are mapped into extremely detailed nested directories:
   * `slurm_evaluation/results/X86/<benchmark>_eval/<benchmark_binary>/<l2_size>/<assoc>/<policy>/<initial_prob>/`
   * The file you fundamentally care about inside those folders is **`stats.txt`**.
